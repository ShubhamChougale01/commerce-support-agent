Commerce Support AI Agent

An AI customer-support agent that reads Zoho Desk tickets for a Medusa.js store and resolves them with a two-pass Claude pipeline — auto-replying when it's confident, escalating to humans when it's not.

General Overview Experienced-dev edition

👋 What Is This Project?

Picture a support inbox for an online store. Customers email things like "where's my order #2?", "please refund this", or "I'm calling my lawyer." This service sits behind that inbox and handles each ticket end-to-end: it reads the message, pulls up the customer's real order data, decides what the ticket actually needs, and either writes a warm on-brand reply (and issues the refund) or hands the ticket to the right human team with a full briefing.

The whole thing is built around two separate Claude calls rather than one mega-prompt. The first call is a cold, JSON-only classifier (what is this, how confident am I, is it fraud?). The second is a warm, brand-voiced reply writer. Splitting "decide" from "speak" is the core design idea — it keeps the decision auditable and the prose human.

It's multi-tenant by design (every client gets its own brand voice/policy via BrandConfig) and safety-first: there's a hard cap on auto-refunds, orders are never auto-cancelled, a quality gate scans every outbound reply for leaked PII, and anything uncertain goes to a person.

So basically — tickets come in from Zoho Desk, Claude classifies then replies (or escalates), and a human only sees the cases that genuinely need judgment.

Pure Python backend, no frontend. Everything talks over HTTP to external SaaS, and each external system is wrapped behind a thin adapter so the business logic never touches a raw API response.

🔄 End-to-End Request Flow

One ticket's journey, from Zoho firing a webhook to either an auto-reply or a human handoff. This is run_agent() in app/agent.py — nine steps, wrapped in a try/except that turns any failure into a system_error escalation (a ticket is never silently dropped).

  Zoho Desk workflow  ──POST /webhook/desk──▶  main.py
   (on Ticket Create)      X-Webhook-Secret        │
                                                   ├─ verify secret (constant-time)
                                                   ├─ idempotency check (Supabase)  ──▶ already seen? 200 & stop
                                                   └─ schedule run_agent() in background, return 200 "accepted"

  run_agent(ticket_id)  ───────────────────────────────────────────────────────────
   1. get_ticket()           fetch + normalize the Zoho ticket
   2. enrich w/ Medusa       order by #id from subject/body, else by requester email
   3. get_ticket_thread()    pull conversation for multi-turn context
   4. classify_ticket()      ── PASS 1 (Sonnet) ──▶  strict JSON: intent, confidence, fraud, action
   5. ESCALATION GATE        escalate flag OR fraud? ─▶ escalate_to_human() & STOP
   6. commerce action        issue_refund (if ≤ ₹2000)  /  cancel_order ─▶ always escalate
   7. generate_reply()       ── PASS 2 (Sonnet) ──▶  warm, brand-voiced email draft
   8. quality_gate           PII / length / overpromise?  fail ─▶ escalate_to_human() & STOP
   9. post_reply() + log     send via Zoho, write analytics row to Supabase
   ─────────────────────────────────────────────────────────────────────────────────
   any exception ───────────▶ escalate_to_human("system_error")

Notice the webhook returns 200 immediately and does the real work in a FastAPI BackgroundTask. Zoho gets a fast ack; Claude calls and API round-trips happen off the request path.

So basically — fetch → enrich → classify → (escalate?) → act → reply → gate → send, and the gate at step 5 and step 8 are where humans get pulled in.

🖥️ Backend Deep Dive

API surface

POST /webhook/desk

The only real entry point. Verifies X-Webhook-Secret (constant-time), de-dupes via Supabase, then dispatches run_agent in the background. Fails closed (503) if the idempotency store is down — better a retry than a duplicate refund.

GET /health

Trivial {"status":"ok"} liveness probe for the platform / load balancer.

The adapter clients (where raw API → clean shape)

Both clients are drop-in replacements for the old commerce/Freshdesk ones — same function signatures, so the orchestrator never changed when the backends were swapped. They normalize messy vendor JSON into the internal contract everything depends on.

ClientWhat it doesWorth knowing
zohodesk.pyToken manager (cache → refresh → retry-on-401), get_ticket, get_ticket_thread, patch_ticket, post_note, post_reply.Strips HTML to text; reply needs explicit from/to so it re-fetches the ticket for the requester email.
medusa.pyOrders by id/email, fulfillment+tracking, issue_refund, get_customer_order_count.Refund is two-step: find the captured payment, then POST a refund against it. Display-id lookup pages newest-first & matches client-side (Medusa silently ignores the display_id filter).

The internal data contract

This is the glue. Memorize the order shape — it shows up in the classifier prompt, the reply prompt, and the escalation note.

// Ticket (from zohodesk.py)
{ id, subject, requester: { email, name }, description_text }

// Order (from medusa.py)
{ order_number, name, email, total_price, fulfillment_status,
  fulfillments: [ { shipment_status, tracking_numbers,
                    tracking_urls, estimated_delivery_at } ] }

Database (Supabase / Postgres) — 3 tables

processed_tickets

Idempotency guard. The webhook inserts a row before doing work; a repeat delivery short-circuits to 200. Primary key is ticket_id.

escalations

The audit + SLA-tracking row written every time a ticket goes to a human: reason, confidence, fraud flags, team, priority, SLA hours, the handoff brief, resolution fields. The watchdog reads this.

tickets_processed

Analytics log written after a reply is auto-sent (intent, confidence, action taken). Note: distinct from processed_tickets despite the near-identical name.

The SLA watchdog

Started inside the FastAPI lifespan context, APScheduler runs sla_watchdog() every 15 minutes. It finds open, un-warned escalations, and once one crosses 75% of its SLA window it fires a Slack warning and flips slack_warned so it doesn't nag.

So basically — two thin clients normalize everything into one order/ticket shape, three Supabase tables handle dedupe + audit + analytics, and a background scheduler nudges humans before they breach SLA.

🤖 AI Orchestration Deep Dive

The "agent" here is deliberately not a tool-calling loop. It's a fixed, auditable pipeline with two distinct Claude calls plus a small one for handoffs. Each call has one job and a tightly scoped prompt.

Pass 1 · Classifier

prompts/classifier.py, Sonnet 4.6, max_tokens=512. JSON-only system prompt → intent, confidence, sentiment, urgency, fraud_flags, refund amount, action, escalate flag. Retries once demanding valid JSON; if still broken, returns a safe escalation dict. It never crashes the pipeline.

Pass 2 · Reply Generator

prompts/reply_generator.py, Sonnet 4.6, max_tokens=800. Builds a brand-aware system prompt from BrandConfig (voice, tone example, refund/exchange policy, sign-off) and writes the customer email. Matches the customer's language; leads with empathy if they're angry.

Handoff Brief

In escalation.py, Haiku 4.5, max_tokens=150. A terse 3-sentence note for the human taking over — problem, what the AI tried, what to do next. Cheap model, deliberately unemotional. Never blocks an escalation if it fails.

Why two passes instead of one?

The decision (structured, must be machine-readable and logged) and the voice (free-form, empathetic, on-brand) are different problems. Separating them means you can log exactly why the agent did something, swap the brand voice per client without touching logic, and unit-test classification independently of prose. The fraud rule is a nice example: the classifier is given CUSTOMER_EMAIL explicitly and only flags a mismatch when it actually has both emails — no guessing.

🛡️ Safety & Guardrails

This is the part that makes it production-shaped rather than a demo. The system is built to fail toward a human, never toward a wrong autonomous action.

Hard escalation gate

Low confidence (<0.75), any fraud flag, legal threats, abuse, "let me talk to a human", or refunds above threshold → straight to the right team. No reply sent.

Refund cap · ₹2000

AUTO_REFUND_LIMIT_INR. Anything above is escalated to the refunds team, never auto-issued.

Never auto-cancel

A cancel_order action always routes to a human (decision D-policy). Cancellations are irreversible enough to warrant a person.

Quality gate

quality_gate.py scans every draft for PII (email, Indian phone, Aadhaar, PAN, card last-4, UPI), length >300 words, and overpromises ("I guarantee"). Fail → escalate with the rejected draft attached.

Idempotency

Webhook de-dupes on ticket_id and fails closed (503) if the store is unreachable — a duplicate refund is worse than a retry.

Catch-all escalation

Any unhandled exception in run_agent becomes a system_error handoff. Even the escalation failing is logged loudly to stderr.

So basically — every risky path (big refund, cancel, uncertainty, leaked PII, a crash) ends with a human holding the ticket, with full context.

⚙️ Local Setup

You can run the app and the test suite without any live accounts — the CI tests mock every external call. You only need real credentials for the integration tests and an actual end-to-end run.

# from the application folder
cd commerce-support-agent

# install deps (use a venv)
pip install -r requirements.txt

# for the no-API-key path: log into the Claude CLI once
claude login

# run the CI suite — 9 tests, fully mocked, must stay green
pytest -m "not integration"

# run the app locally (then GET http://localhost:8000/health)
uvicorn app.main:app --reload

# live API smoke tests (needs a full .env)
pytest -m integration

# build the container (verified working)
docker build -t support-agent .
Windows console tip: prefix Python runs with PYTHONIOENCODING=utf-8 when output may contain ₹ — the cp1252 console chokes on it otherwise.

🎯 Your Personalized Roadmap

You're experienced across the stack and want the full picture, so this is an orientation path, not a tutorial. The fastest way to own this codebase is to trace one ticket all the way through, then poke at the edges.

Hour 1 — Get it green and read the orchestrator
  • Clone, pip install -r requirements.txt, run pytest -m "not integration" — confirm 9 green.
  • Read app/agent.py top to bottom. The 9 numbered steps are the architecture; everything else is a helper.
  • Open tests/test_e2e.py alongside it — the mocks show you exactly what each step expects and returns.
Day 1 — Understand the two passes & the contract
  • Read both prompts in prompts/. Note the strict JSON schema in the classifier and the brand injection in the reply generator.
  • Memorize the internal Ticket/Order shapes — they're the contract between adapters and logic.
  • Skim _llm_client.py so you understand the API-key vs. subscription-shim switch before you wonder why there's no key.
Day 2 — The adapters & the safety net
  • Read zohodesk.py (token refresh + normalization) and medusa.py (two-step refund + the display-id paging quirk).
  • Read escalation.py end-to-end — it's the most "wiring-heavy" file and where most real-world bugs will live.
  • Read quality_gate.py and config.py's ROUTING table — they define what's allowed out the door and where escalations land.
Day 3+ — Run it live & find the open threads
  • Bring up local Medusa via docker-compose.medusa.yml, point a real .env at it, and run pytest -m integration.
  • Check BLOCKERS.md for the live go-live checklist — that's the current source of truth on what's left.
  • Pick up a known TODO to get your hands dirty: tokens_used is logged as None, the Supabase-backed BrandConfig lookup is stubbed, and COMPETITOR_BRANDS is empty.
